## Задача 1: Реализация автоматического переподключения SSE

### Проблема
В обработчике ошибок SSE (Server-Sent Events) есть пустая функция setTimeout для реконнекта. При обрыве соединения с сервером (например, при нестабильном интернете или перезапуске backend) подключение не восстанавливается автоматически.

### Текущее поведение
При возникновении ошибки SSE:
1. Подключение закрывается
2. Состояние isConnected меняется на false
3. Вызывается setTimeout с пустой функцией
4. Реконнект НЕ происходит
5. Пользователь перестаёт получать уведомления до перезагрузки страницы

### Причина
**Файл:** `frontend/src/lib/contexts/SSEContext.tsx`  
**Строки:** 56-58

В обработчике ошибки есть комментарий о необходимости реализации, но логика отсутствует.

### Что нужно сделать

1. **Создать систему переподключения с экспоненциальным backoff:**
   - Первая попытка переподключения через 1 секунду
   - Вторая попытка через 2 секунды
   - Третья попытка через 4 секунды
   - Четвёртая попытка через 8 секунд
   - Максимальная задержка - 30 секунд

2. **Добавить счётчик попыток:**
   - Ограничить максимальное количество попыток (например, 10)
   - После исчерпания попыток показать пользователю уведомление
   - Возможность сбросить счётчик при успешном подключении

3. **Добавить состояние для отслеживания:**
   - Текущее количество попыток переподключения
   - Статус: подключено / переподключение / ошибка
   - Опционально: время следующей попытки

4. **Улучшить UX:**
   - Показывать индикатор "Переподключение..." во время попыток
   - Уведомить пользователя если переподключение не удалось
   - Сбрасывать счётчик попыток при успешном подключении

5. **Логирование:**
   - Логировать каждую попытку переподключения
   - Логировать успешные подключения
   - Логировать финальную неудачу (если все попытки исчерпаны)

### Ожидаемый результат
- При временной потере соединения SSE автоматически переподключается
- Пользователь продолжает получать уведомления после восстановления связи
- Нет необходимости перезагружать страницу вручную
- Разумное использование ресурсов (экспоненциальный backoff)
- Информативные сообщения для пользователя о статусе подключения

### Критерии приемки
- ✅ При обрыве соединения происходит автоматическое переподключение
- ✅ Используется экспоненциальный backoff (1, 2, 4, 8... до 30 сек)
- ✅ Есть лимит попыток переподключения
- ✅ Счётчик сбрасывается при успешном подключении
- ✅ Пользователь видит статус подключения
- ✅ После исчерпания попыток показывается понятное сообщение

## Задача 2: Безопасная передача токена в SSE

### Проблема
JWT токен авторизации передаётся в URL query параметрах при подключении к SSE endpoint. Это создаёт несколько угроз безопасности:
- Токен попадает в логи веб-сервера (обычно URL логируются полностью)
- Токен сохраняется в истории браузера
- Токен может утечь через Referer header при переходах
- Не соответствует best practices безопасности для JWT токенов

### Текущее поведение
**Файл:** `frontend/src/lib/contexts/SSEContext.tsx`  
**Строка:** 25

Токен добавляется как query параметр:
- URL выглядит как: `/api/notifications/stream?token=eyJhbGc...`
- Токен виден в логах сервера
- Токен может быть перехвачен

### Причина
EventSource (стандартный API для SSE) не поддерживает custom headers, поэтому токен приходится передавать через URL.

### Что нужно сделать

**Вариант A: Использование Cookies (рекомендуется для SSE)**

1. **На backend:**
   - Настроить эндпоинт SSE для чтения токена из HttpOnly cookie
   - Установить cookie при успешном логине
   - Убедиться что cookie отправляется с SSE запросом

2. **На frontend:**
   - Убрать передачу токена в URL query параметрах
   - Полагаться на автоматическую отправку cookies браузером
   - Убедиться что cookie включены в credentials

3. **Настройка безопасности cookie:**
   - HttpOnly: true (защита от XSS)
   - Secure: true (только HTTPS в production)
   - SameSite: Strict или Lax
   - Path: /api (ограничить область действия)

**Вариант B: Миграция на WebSocket (если SSE недостаточно)**

1. Заменить EventSource на WebSocket
2. WebSocket поддерживает отправку токена через headers или через первое сообщение
3. Реализовать handshake с передачей токена
4. Настроить аутентификацию на backend

**Вариант C: Токен в последнем сегменте URL (временное решение)**

1. Вместо `/stream?token=XXX` использовать `/stream/XXX`
2. Это немного безопаснее query params (не попадает в Referer)
3. Но всё равно попадает в логи сервера

### Рекомендуемый вариант
**Вариант A (Cookies)** - наиболее безопасный и простой для SSE.

### Ожидаемый результат
- JWT токен НЕ передаётся в URL
- Токен НЕ попадает в логи веб-сервера
- Токен НЕ сохраняется в истории браузера
- Токен НЕ утекает через Referer header
- SSE подключение остаётся рабочим и аутентифицированным
- Соответствие стандартам безопасности

### Критерии приемки
- ✅ Токен НЕ присутствует в URL при подключении к SSE
- ✅ SSE подключение успешно аутентифицируется
- ✅ Backend корректно валидирует токен из cookie/headers
- ✅ Уведомления продолжают работать
- ✅ При logout cookie удаляется или инвалидируется
- ✅ В production используется Secure cookie (HTTPS)

## Задача 3: Оптимизация загрузки данных в ExpenseDetail

### Проблема
Страница деталей одной заявки загружает ВСЕ заявки из базы данных (лимит 1000 записей) вместо того, чтобы запросить только одну нужную запись по ID. Это приводит к:
- Медленной загрузке страницы (особенно при плохом интернете)
- Ненужному трафику (загружается ~1000 записей вместо 1)
- Увеличенной нагрузке на backend и базу данных
- Плохому пользовательскому опыту
- Неэффективному использованию React Query кэша

### Текущее поведение
**Файл:** `frontend/src/pages/ExpenseDetail.tsx`  
**Строка:** 79

Компонент делает запрос на получение всех заявок, затем ищет нужную по ID в массиве на клиенте.

### Причина
Отсутствует API endpoint для получения одной заявки по ID, поэтому используется обходной путь через получение всех записей.

### Что нужно сделать

**Backend часть (обязательно):**

1. **Создать новый API endpoint:**
   - Путь: `GET /api/expenses/{id}`
   - Параметры: id заявки в URL
   - Возвращает: одну заявку или 404 если не найдена
   - Проверка авторизации: пользователь имеет доступ к этой заявке

2. **Оптимизировать запрос к базе:**
   - Использовать WHERE clause для фильтрации по ID
   - Загружать только необходимые поля
   - Включить связанные данные через JOIN (проект, автор)
   - Добавить индекс на поле id если отсутствует

**Frontend часть:**

1. **Создать новый метод в сервисе:**
   - Файл: `frontend/src/lib/services/expenses.ts`
   - Добавить метод `getExpenseById(id: string)`
   - Использовать новый API endpoint

2. **Обновить компонент ExpenseDetail:**
   - Заменить запрос `getExpenses({ limit: 1000 })`
   - Использовать `getExpenseById(id)`
   - Обновить queryKey для корректного кэширования

3. **Настроить React Query:**
   - Использовать отдельный queryKey: `["expense", id]`
   - Настроить staleTime для кэширования
   - Добавить prefetch при наведении на карточку заявки (опционально)

4. **Обработка ошибок:**
   - Показывать 404 если заявка не найдена
   - Показывать loader во время загрузки
   - Обрабатывать ошибки сети

### Ожидаемый результат
- Быстрая загрузка страницы деталей заявки
- Минимальный трафик (загружается только 1 запись)
- Снижение нагрузки на backend
- Улучшенный UX (особенно при медленном интернете)
- Эффективное использование кэша React Query
- Правильная работа при большом количестве заявок

### Критерии приемки
- ✅ Backend endpoint `GET /api/expenses/{id}` создан и работает
- ✅ ExpenseDetail использует новый endpoint
- ✅ Загружается только одна запись (проверить в DevTools Network)
- ✅ Страница загружается быстрее (измерить время)
- ✅ 404 ошибка обрабатывается корректно
- ✅ React Query кэширует результат правильно
- ✅ Не сломан функционал истории изменений и других данных

## Задача 4: Замена non-null assertion на безопасную проверку

### Проблема
В коде используется non-null assertion operator (`!`) в месте, где значение теоретически может быть undefined. TypeScript не будет предупреждать об ошибке, и это может привести к runtime ошибке при неожиданных условиях.

### Текущее поведение
**Файл:** `frontend/src/pages/Applications.tsx`  
**Строка:** 59

Используется `expensesPage!.items` - предполагается что expensesPage всегда определён.

### Причина
Хотя выше есть проверка `if (expensesPage?.items)`, использование `!` всё равно небезопасно и считается антипаттерном.

### Что нужно сделать

1. **Заменить non-null assertion на optional chaining:**
   - Использовать `expensesPage?.items.filter(...)` вместо `expensesPage!.items.filter(...)`
   - Добавить fallback значение `|| []` на случай undefined

2. **Добавить явную проверку типа:**
   - Убедиться что все пути кода обрабатывают undefined
   - Использовать TypeScript narrowing для безопасности типов

### Ожидаемый результат
- Код более безопасен и предсказуем
- Нет потенциальных runtime ошибок
- TypeScript корректно проверяет типы
- Соответствие best practices

### Критерии приемки
- ✅ Non-null assertion operator удалён
- ✅ Используется optional chaining с fallback
- ✅ TypeScript не показывает ошибок
- ✅ Функционал работает без изменений

## Задача 5: Закрытие SSE подключения при logout

### Проблема
SSE подключение создаётся один раз при монтировании компонента и не следит за состоянием авторизации. После logout подключение остаётся открытым и продолжает попытки получить уведомления с невалидным токеном.

### Текущее поведение
**Файл:** `frontend/src/lib/contexts/SSEContext.tsx`  
**Строки:** 15-18, 65

useEffect создаёт подключение с пустым массивом зависимостей `[]`, поэтому:
1. Подключение создаётся один раз
2. При logout токен удаляется
3. Подключение остаётся открытым
4. Backend получает запросы с невалидным токеном
5. Генерируются лишние ошибки в логах

### Причина
Эффект не следит за изменением токена в localStorage.

### Что нужно сделать

**Вариант A: Следить за токеном (простое решение)**

1. Добавить состояние для хранения токена
2. Обновлять состояние при изменении localStorage
3. Добавить токен в массив зависимостей useEffect
4. При изменении токена - пересоздавать подключение

**Вариант B: Слушатель события logout (более явное решение)**

1. Создать custom event для logout
2. Диспатчить событие при вызове store.logout()
3. Слушать событие в SSEContext
4. Закрывать подключение при получении события

**Вариант C: Использовать Context для авторизации**

1. Создать AuthContext с состоянием isAuthenticated
2. Использовать этот контекст в SSEContext
3. Подписаться на изменения isAuthenticated
4. Управлять подключением на основе состояния

### Рекомендуемый вариант
**Вариант B** - наиболее явный и понятный.

### Ожидаемый результат
- При logout SSE подключение закрывается
- Нет лишних запросов к backend с невалидным токеном
- Нет ошибок в логах backend
- При повторном login подключение создаётся заново
- Чистая архитектура с явным управлением жизненным циклом

### Критерии приемки
- ✅ При logout SSE подключение закрывается
- ✅ Нет запросов к SSE endpoint после logout (проверить в Network)
- ✅ При повторном login подключение создаётся заново
- ✅ Уведомления работают корректно после re-login

## Задача 6: Оптимизация стратегии обновления данных (polling vs SSE)

### Проблема
В компоненте Applications данные обновляются автоматически каждые 10 секунд (polling), при этом в приложении также используется SSE для получения уведомлений в реальном времени. Это создаёт избыточность:
- Двойная нагрузка на backend (polling + SSE)
- Лишний трафик даже при отсутствии изменений
- Расход батареи на мобильных устройствах
- Потенциальные конфликты обновлений

### Текущее поведение
**Файл:** `frontend/src/pages/Applications.tsx`  
**Строка:** 40

Установлен `refetchInterval: 10000` - автоматический рефетч каждые 10 секунд независимо от активности пользователя.

### Причина
Polling был добавлен как простой способ держать данные актуальными до реализации SSE.

### Что нужно сделать

**Вариант A: Использовать только SSE (рекомендуется)**

1. **Убрать polling полностью:**
   - Удалить `refetchInterval: 10000`
   - Полагаться только на SSE уведомления

2. **Улучшить SSE для обновления данных:**
   - При получении уведомления вызывать `queryClient.invalidateQueries()`
   - Обновлять только затронутые ресурсы
   - Оптимистичные обновления для мгновенного отклика

3. **Fallback для надёжности:**
   - Добавить кнопку "Обновить" для ручного рефетча
   - Обновлять при возвращении на вкладку (refetchOnWindowFocus)
   - Рефетч при ошибке SSE подключения

**Вариант B: Умный polling**

1. **Увеличить интервал:**
   - Изменить с 10 секунд на 30-60 секунд
   - Использовать как резервный механизм к SSE

2. **Отключать при неактивности:**
   - Проверять видимость вкладки через Page Visibility API
   - Останавливать polling когда вкладка неактивна
   - Возобновлять при возвращении

3. **Адаптивный интервал:**
   - Короткий интервал при активной работе (15 сек)
   - Длинный интервал при отсутствии активности (60 сек)
   - Отключение после 5 минут неактивности

### Рекомендуемый вариант
**Вариант A** - полагаться на SSE, убрать дублирование.

### Ожидаемый результат
- Снижение нагрузки на backend
- Уменьшение сетевого трафика
- Экономия батареи на мобильных устройствах
- Данные обновляются моментально через SSE
- Нет избыточных запросов
- Надёжный fallback для случаев проблем с SSE

### Критерии приемки
- ✅ Polling либо убран, либо оптимизирован
- ✅ Данные обновляются через SSE
- ✅ Есть механизм ручного обновления
- ✅ Функционал работает без ухудшения UX
- ✅ Снижение количества запросов (измерить в DevTools)
- ✅ Есть fallback на случай проблем с SSE

## Риски и ограничения

### Возможные риски:

1. **Задача 2 и 3 требуют изменений backend:**
   - Нужна координация с backend разработчиком
   - Может потребоваться время на backend разработку
   - Требуется деплой backend

2. **SSE реконнект может увеличить нагрузку:**
   - При массовом обрыве соединений много одновременных реконнектов
   - Решение: randomize задержки реконнекта

3. **Отключение polling может ухудшить UX при проблемах SSE:**
   - Если SSE не работает, данные не обновляются
   - Решение: добавить кнопку ручного обновления

### Ограничения:

- Нельзя изменять структуру данных в localStorage
- Нужно сохранить обратную совместимость
- Backend API должен поддерживать новые endpoints
