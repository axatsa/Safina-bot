# Аудит Safina Finance Bot — детальный разбор проблем

---

## 🔴 Критические — сломается в продакшене

---

### 1. `decisions.py` — падение при решении CEO (DetachedInstanceError)

**Файл:** `backend/app/services/bot/handlers/decisions.py`, функции `handle_approve_ceo` и `handle_reject_ceo`

**Что происходит:**
Объект `expense` загружается из БД внутри блока `with db:`, но после выхода из блока
SQLAlchemy закрывает сессию и "отвязывает" объект. Когда дальше по коду вызывается
`expense.request_id` или `expense.total_amount` для отправки уведомления — SQLAlchemy
пытается снова сходить в БД через уже закрытую сессию и выбрасывает
`DetachedInstanceError`.

**Последствие:** каждый раз когда CEO нажимает "Одобрить" или "Отклонить" — бот падает с
ошибкой, уведомления в Safina и CFO не доходят, в логах ошибка.

**Как выглядит сейчас (неправильно):**
```python
with next(database.get_db()) as db:
    expense = db.query(...).first()
    crud.update_expense_status(...)
# ← здесь сессия уже закрыта, expense "отвязан"

asyncio.create_task(send_ceo_decision_notification(
    admin_id, expense.request_id, ...  # ← ПАДАЕТ: DetachedInstanceError
))
```

**Как исправить:** сохранить нужные поля в локальные переменные внутри блока `with db:`,
до его закрытия:
```python
with next(database.get_db()) as db:
    expense = db.query(...).first()
    crud.update_expense_status(...)
    # сохраняем нужное ДО выхода из блока
    req_id = expense.request_id
    amount = float(expense.total_amount)
    currency = expense.currency

# теперь безопасно
asyncio.create_task(send_ceo_decision_notification(admin_id, req_id, amount, currency, True))
```

### 2. `decisions.py` — `asyncio.create_task()` без гарантированного event loop

**Файл:** `backend/app/services/bot/handlers/decisions.py`

**Что происходит:**
`asyncio.create_task()` требует чтобы в момент вызова уже существовал запущенный event loop.
В aiogram callback-хендлерах это обычно есть, но при перезапуске бота через watchdog или в
тестовом окружении это не гарантировано — `create_task` выбрасывает
`RuntimeError: no running event loop` и уведомления теряются.

**Как исправить:** заменить `asyncio.create_task(...)` на `await asyncio.gather(...)` прямо
внутри async-хендлера — это надёжнее и проще:
```python
# вместо:
asyncio.create_task(send_ceo_decision_notification(admin_id, ...))

# использовать:
await asyncio.gather(
    send_ceo_decision_notification(admin_id, ...),
    *[send_ceo_decision_notification(cfo_id, ...) for cfo_id in cfo_ids]
)
```

### 3. `.env.example` содержит нераскрытые маркеры конфликта Git

**Файл:** `.env.example` в корне проекта

**Что происходит:**
В файле остались маркеры незавершённого слияния веток (`<<<<<<< HEAD`, `=======`,
`>>>>>>> eee1b37...`). Файл выглядит так:
```
<<<<<<< HEAD
DB_USER=safina
BOT_TOKEN=your_token
=======
VITE_API_URL=/api
>>>>>>> eee1b37b
```
Если разработчик или DevOps скопирует этот файл как `.env` и запустит приложение — FastAPI
не сможет распознать переменные из-за мусорных символов, бот не запустится, БД не подключится.

**Как исправить:** вручную открыть `.env.example`, убрать все маркеры слияния и оставить
один корректный вариант файла с комментариями.

### 4. Аналитика и графики смешивают UZS и USD

**Файл:** `backend/app/api/analytics.py`

**Что происходит:**
В коде строится график расходов по дням — но суммы складываются без учёта валюты:
```python
amount = float(expense.total_amount)  # 100 USD и 500 000 UZS складываются как 500 100
timeline_data[date_str]["expenses"] += amount
```
На дашборде финансисты видят графики где 100 долларов и 1 300 000 сумов показаны как
1 300 100 — цифра финансово бессмысленная.

**Как исправить:** при суммировании конвертировать USD в UZS через сохранённый курс
(`usd_rate` — из отдельного ТЗ по курсам валют).

## 🔴 Дыры в безопасности

---

### 5. Удаление проектов и сотрудников без проверки токена

**Файлы:** `backend/app/api/team.py`, `backend/app/api/projects.py`

**Что происходит:**
Следующие эндпоинты **не требуют авторизации** — нет `Depends(auth.get_current_user)`:
- `DELETE /api/team/{member_id}` — удалить любого сотрудника
- `DELETE /api/projects/{project_id}` — удалить проект (CASCADE удалит ВСЕ заявки проекта)
- `POST /api/projects/{id}/members/{mid}` — добавить кого угодно в проект
- `DELETE /api/projects/{id}/members/{mid}` — удалить участника из проекта

Любой человек, знающий URL API (из логов браузера, Postman), может удалить проект Thompson
Land со всеми заявками одним HTTP-запросом без логина.

**Как исправить:** добавить `current_user: models.TeamMember = Depends(auth.get_current_user)`
в параметры каждого из этих роутеров и проверку `is_admin(current_user)`.

### 6. JWT токен действует 1 год без возможности отозвать

**Файл:** `backend/app/core/auth.py`

**Что происходит:**
```python
ACCESS_TOKEN_EXPIRE_MINUTES = 60 * 24 * 365  # 1 год
```
Токен выдаётся на 365 дней и нет механизма его отзыва. Если сотрудник уволился, его токен
продолжает работать ещё год — он может заходить в систему и смотреть все заявки. Если токен
утёк через скриншот переписки или логи — злоумышленник имеет доступ на год.

**Как исправить:** уменьшить время жизни до 24 часов и добавить refresh token с хранением в БД
(тогда при увольнении — удаляем refresh token, access token перестаёт обновляться).

### 7. Дефолтные пароли в `seed.py` и `auth.py`

**Файлы:** `backend/app/db/seed.py`, `backend/app/core/auth.py`

**Что происходит:**
Если при деплое не задать переменные окружения `SF_PASSWORD`, `CEO_PASSWORD`, `ADMIN_PASSWORD`
— система молча создаёт аккаунты с паролями `s_admin123`, `g_admin123`, `admin123`. Это
случается чаще чем кажется: при первом запуске на новом сервере, при сбросе `.env` файла.
CFO (Носиров) и CEO с такими паролями видны любому кто попробует стандартные комбинации.

**Как исправить:** убрать дефолтные значения из кода, при отсутствии переменной — не создавать
пользователя вообще, а логировать критическую ошибку `"SF_PASSWORD not set — seeding skipped"`.

### 8. CORS открыт для всех доменов (`*`)

**Файл:** `backend/main.py`

**Что происходит:**
В коде есть список разрешённых доменов (`finance.thompson.uz`, `localhost:3000` и т.д.), но
вместо него в middleware стоит `allow_origins=["*"]` — запросы принимаются с абсолютно любого
сайта. Это значит любая страница в интернете может делать запросы к API от имени
авторизованного пользователя через его браузер (CSRF-атака).

**Как исправить:** заменить `allow_origins=["*"]` на уже написанный список `origins`, который
есть в том же файле но почему-то не используется.

### 9. `web-submit` принимает `chat_id` из тела запроса

**Файл:** `backend/app/api/expenses.py`, эндпоинт `POST /expenses/web-submit`

**Что происходит:**
```python
chat_id = data.get("chat_id")  # пользователь сам говорит кто он
user = db.query(...).filter(telegram_chat_id == chat_id).first()
```
Сервер доверяет `chat_id` из тела запроса, не проверяя токен. Любой авторизованный
пользователь может поставить чужой `chat_id` и создать заявку от имени другого сотрудника.

**Как исправить:** `chat_id` должен браться из JWT токена (`current_user.telegram_chat_id`),
а не из тела запроса — тогда нельзя подделать отправителя.

## 🟡 Потеря данных при эксплуатации

### 10. FSM состояния бота хранятся в памяти (`MemoryStorage`)

**Файл:** `backend/app/services/bot/main.py`

**Что происходит:**
```python
storage = MemoryStorage()
```
Все текущие диалоги пользователей (кто на каком шаге FSM, что уже ввёл) живут только в
оперативной памяти процесса. При любом рестарте бота — деплое, крэше, плановом обновлении —
все незаконченные диалоги обнуляются. Пользователь, который дошёл до шага 5 из 8 при
заполнении заявки, после рестарта получит пустое состояние и не поймёт что произошло.

**Как исправить:** переключить на `RedisStorage` из пакета `aiogram-redis` — состояния
сохраняются в Redis и переживают перезапуск. В коде уже есть комментарий об этом, нужно
просто реализовать.

### 11. `Base.metadata.create_all()` не применяет изменения в существующих таблицах

**Файл:** `backend/main.py`, функция `init_db_with_retry`

**Что происходит:**
```python
Base.metadata.create_all(bind=engine)
```
Эта команда создаёт таблицы если их нет, но **не изменяет уже существующие**. Когда добавишь
новую колонку в модель (например `usd_rate` для курса валют), задеплоишь — колонка появится
в Python-классе, но не в реальной БД. Все запросы к этой колонке начнут падать с ошибкой
`column does not exist`.

**Как исправить:** настроить Alembic — инструмент миграций для SQLAlchemy. При каждом деплое
запускать `alembic upgrade head`, который применяет только новые изменения.

### 12. Загруженные файлы чеков хранятся по относительному пути без volume

**Файл:** `backend/app/services/refund/service.py`

**Что происходит:**
```python
upload_dir = "uploads/receipts"  # относительный путь
```
В Docker-контейнере файлы сохраняются внутри контейнера по относительному пути. При каждом
пересборке образа или перезапуске контейнера без `volume` — все загруженные фото чеков
безвозвратно удаляются. Финансисты откроют старую заявку, нажмут "посмотреть чек" — файла нет.

**Как исправить:** путь загрузок брать из переменной окружения `UPLOAD_DIR` и в
`docker-compose.yml` монтировать эту директорию как постоянный volume:
```yaml
volumes:
  - ./uploads:/app/uploads
```

### 13. Нет истории изменений статусов заявки

**Вся система обработки заявок**

**Что происходит:**
В таблице `expense_requests` хранится только текущий статус. Когда заявка проходит путь
`request → review → pending_senior → approved_senior → confirmed` — все промежуточные
состояния теряются. Невозможно ответить на вопросы: "Почему заявка была отклонена в
прошлый раз?", "Кто и когда переводил статус?", "Сколько дней заявка ждала у CFO?".

**Как исправить:** создать таблицу `expense_status_history` с полями
`(expense_id, from_status, to_status, changed_by_id, changed_at, comment)` и писать туда
запись при каждом изменении статуса в `crud.update_expense_status`.

### 14. Физическое удаление сотрудника не лишает его доступа

**Файл:** `backend/app/api/team.py`

**Что происходит:**
```python
db.delete(user)
```
При удалении сотрудника из БД его JWT-токен продолжает быть валидным до истечения срока
(сейчас — 1 год). Уволенный сотрудник с сохранённым токеном в браузере может продолжать
заходить в систему и видеть все заявки.

**Как исправить:** вместо физического удаления переводить статус в `blocked`
(`user.status = "blocked"`) — в `get_current_user` уже есть проверка статуса, которая
заблокирует доступ немедленно. Это также сохраняет исторические данные о том кто создавал
заявки.

### 15. Нет защиты от случайного дублирования заявки

**Файл:** `backend/app/db/crud.py`, функция `create_expense_request`

**Что происходит:**
Если пользователь в боте быстро нажмёт "Готово" дважды, или сеть "моргнёт" и форма
отправит данные повторно — создадутся две одинаковые заявки с разными `request_id`.
Финансисты увидят дубликат, начнут разбираться, потратят время на его отклонение.

**Как исправить:** добавить временную блокировку на 10 секунд после создания заявки
по ключу `user_id` (через Redis или простую in-memory проверку), либо на фронтенде
блокировать кнопку после первого нажатия до получения ответа.

### 16. Денормализованные поля — осознанное решение, но нужно документировать

**Файл:** `backend/app/db/models.py`

**Что происходит:**
При создании заявки копируются ФИО, должность сотрудника и название проекта:
```python
created_by = Column(String)       # "Баходиров Баходир"
project_name = Column(String)     # "Thompson Land"
```
Это правильный паттерн для финансовых систем — исторические записи не должны меняться.
Но если этого не знать, можно случайно написать код который ожидает актуальных данных из
этих полей, а они отражают момент создания заявки.

**Что сделать:** добавить комментарий в модели: `# Denormalized at creation time. Do not update.`

### 17. Каждое уведомление создаёт новый `Bot()` инстанс

**Файл:** `backend/app/services/bot/notifications.py`

**Что происходит:**
```python
async def _send_message(chat_id, text, ...):
    bot = Bot(token=os.getenv("BOT_TOKEN"))  # новый объект и HTTP-сессия на каждый вызов
    try:
        await bot.send_message(...)
    finally:
        await bot.session.close()  # сразу закрываем
```
При каждом уведомлении открывается и закрывается HTTP-сессия к Telegram API. При 20+
заявках одновременно это создаёт лишнюю нагрузку и замедляет отправку. Кроме того, при
быстром потоке уведомлений есть риск превысить rate limit Telegram (30 сообщений/сек).

**Как исправить:** создать один глобальный `Bot` инстанс при старте приложения и
переиспользовать его во всех функциях уведомлений.

### 18. Потенциальные N+1 запросы при работе со связанными объектами

**Файл:** `backend/app/api/analytics.py` и другие

**Что происходит:**
SQLAlchemy по умолчанию загружает связанные объекты "лениво" (lazy loading) — каждый раз
когда где-то в коде обращаются к `expense.project` или `expense.created_by_user`, идёт
отдельный SQL-запрос. Если в цикле обрабатываются 100 заявок и у каждой читается
`expense.project.name` — это 100 дополнительных запросов к БД вместо одного JOIN.

**Как исправить:** в запросах где нужны связанные данные добавлять
`.options(joinedload(models.ExpenseRequest.project))` — это загрузит всё одним запросом.

### 19. Нет лимита на количество позиций в заявке

**Файл:** `backend/app/db/schemas.py`

**Что происходит:**
```python
items: List[ExpenseItemSchema]  # без ограничения
```
Технически можно создать заявку с 10 000 позиций — всё это сохранится в одну JSON-колонку
в БД. При экспорте такой заявки в Excel или при её отображении на фронтенде страница
зависнет или упадёт. Это маловероятно при нормальном использовании, но возможно при
ошибке или намеренно.

**Как исправить:** добавить ограничение в Pydantic схему:
```python
items: List[ExpenseItemSchema] = Field(..., max_items=50)
```

### 20. `/api/health` не проверяет реальное состояние системы

**Файл:** `backend/main.py`

**Что происходит:**
```python
@app.get("/api/health")
async def health_check():
    return {"status": "ok", "database": "connected"}  # всегда "connected"
```
Эндпоинт всегда возвращает `ok` независимо от того работает ли БД. Если подключить этот
URL в мониторинг (UptimeRobot, Grafana) — при падении PostgreSQL мониторинг будет
сообщать что всё хорошо, а реальные пользователи не смогут войти.

**Как исправить:** выполнять реальный запрос `SELECT 1` к БД внутри эндпоинта и возвращать
`503` если запрос не прошёл.

### 21. Нет UI для архивирования заявок

**Файл:** `backend/app/api/expenses.py` и фронтенд

**Что происходит:**
Статус `archived` есть в системе и заархивированные заявки исключены из экспортов, но
перевести заявку в архив можно только прямым вызовом API через Postman или curl. В
интерфейсе кнопки "Архивировать" нет. Со временем список подтверждённых заявок будет
бесконечно расти, замедлять запросы и засорять интерфейс.

**Как исправить:** добавить кнопку "Архивировать" в карточку заявки на фронтенде для
пользователей с ролью `admin` — вызывает существующий `PATCH /{id}/status` со статусом
`archived`.

### 22. Пагинация ограничена 100 записями без индикатора

**Файл:** `backend/app/api/expenses.py`

**Что происходит:**
```python
def read_expenses(..., limit: int = 100, ...):
```
API возвращает максимум 100 заявок. При активной работе — это месяц-полтора, после чего
часть данных просто не будет загружаться в интерфейс. Если фронтенд не показывает
предупреждение об обрезании списка — финансисты будут думать что видят все заявки, хотя
видят только последние 100.

**Как исправить:** убедиться что фронтенд передаёт параметры `skip` и `limit` и реализует
постраничную навигацию или бесконечный скролл.
