# ТЗ: Система курсов валют — Thompson Finance Bot

---

## Текущая проблема

В коде экспорта (`/export-xlsx`) сумма итого считается формулой `=SUM(K2:K...)` по колонке
`total_amount` **без разбора валюты**. Если в одном отчёте есть заявки в UZS и в USD — итог
**неверный**. Это финансовая ошибка.

Также `ExpenseRequest` не хранит курс на момент создания — восстановить его потом невозможно.

---

## Анализ требований

| Сценарий | Что нужно | Вывод |
|---|---|---|
| Активная заявка (до 3 дней) | Актуальный курс — финансисты обработают за 3 дня | Брать текущий курс ЦБ при создании |
| Архив | Исторически верный курс | Сохранять курс в момент создания |
| Месячный отчёт | Средний курс за месяц | Считается **автоматически** как среднее сохранённых курсов |

**Вывод:** самый правильный и простой подход — **сохранить курс ЦБ один раз при создании
заявки**. Тогда:
- Архив всегда точный — курс на тот день зафиксирован
- Для месячного отчёта: среднее всех сохранённых курсов за месяц — это и есть средний курс
  месяца, без ручного ввода
- Активные заявки — курс актуальный (создана сегодня = курс сегодня)

Никакого ручного ввода средних курсов не нужно.

---

## Откуда брать курс

**ЦБ Узбекистана (CBU)** публикует официальные курсы через открытый API:

```
GET https://cbu.uz/ru/arkhiv-kursov-valyut/json/USD/
```

Возвращает массив, первый элемент — актуальный курс:
```json
[{ "Ccy": "USD", "Rate": "12750.54", "Date": "14.03.2026" }]
```

Это официальный источник, бесплатный, без ключей. Именно его используют для бухгалтерии в
Узбекистане.

## Изменения в БД

### Новая колонка в `ExpenseRequest`

```
usd_rate  — Numeric(18, 6), nullable
```

Логика:
- Если валюта заявки `UZS` — `usd_rate = null` (не нужен, всё уже в сумах)
- Если валюта `USD` — `usd_rate = курс ЦБ на момент создания`

> Почему не хранить для UZS тоже? Потому что `usd_rate` означает "по какому курсу считать эту
> заявку". Для UZS курс 1:1, хранить бессмысленно. В расчётах: если `usd_rate` is null → сумма
> уже в UZS, конвертация не нужна.

### Формула конвертации

```
сумма_в_uzs = total_amount × usd_rate    (если валюта USD)
сумма_в_uzs = total_amount               (если валюта UZS)
```

## Что меняется в коде

### 1. `crud.py` → `create_expense_request`

При создании заявки:
1. Если `currency == "USD"` — делаем запрос к CBU API, получаем курс
2. Сохраняем курс в `db_expense.usd_rate`
3. Если CBU недоступен — логируем ошибку, сохраняем `usd_rate = null` (не блокировать создание
   заявки из-за недоступности API)

### 2. `models.py` → `ExpenseRequest`

Добавить колонку:
```python
usd_rate = Column(Numeric(precision=18, scale=6), nullable=True)
# Курс USD/UZS на момент создания заявки. null для UZS-заявок.
```

Новая алембик-миграция.

### 3. `api/expenses.py` → экспорт CSV и XLSX

Добавить в каждую строку:
- Колонку **"Курс USD"** — значение `usd_rate` (пустая ячейка для UZS заявок)
- Колонку **"Сумма в UZS"** — рассчитанное значение

В XLSX итоговая строка:
- `=SUM(...)` считается только по колонке **"Сумма в UZS"** — это всегда будет корректно
- Старая колонка "Общая сумма" остаётся для сведений (в оригинальной валюте)

### 4. `api/expenses.py` → ответ API для детальной заявки

В схеме `ExpenseRequestSchema` добавить поле `usd_rate` — чтобы фронтенд мог показывать курс
в карточке заявки.

## Месячный отчёт

Финансисты хотят "средний курс за месяц". Теперь это считается **автоматически**:

```sql
SELECT
    DATE_TRUNC('month', date) AS month,
    AVG(usd_rate)             AS avg_usd_rate,
    SUM(
        CASE WHEN currency = 'USD'
             THEN total_amount * usd_rate
             ELSE total_amount
        END
    ) AS total_uzs
FROM expense_requests
WHERE status IN ('confirmed', 'archived')
GROUP BY month
ORDER BY month
```

Этот запрос даёт и средний курс за месяц, и итоговую сумму в UZS.

## Крайние случаи

| Ситуация | Поведение |
|---|---|
| CBU API недоступен при создании | Сохранить `usd_rate = null`, не блокировать заявку. В экспорте показывать "курс не сохранён", в итог не включать эту строку |
| Старые заявки без `usd_rate` | `usd_rate = null` — в экспорте показывать оригинальную сумму и валюту, в итог в UZS не включать. Финансисты могут вручную проставить через admin-панель |
| Заявка в USD изменила статус позже | Курс не пересчитывается. Зафиксирован на момент создания — это правильно |
| В будущем появится третья валюта | Колонка `usd_rate` становится `exchange_rate`, добавляется `base_currency = "UZS"` — схема выдержит расширение |
